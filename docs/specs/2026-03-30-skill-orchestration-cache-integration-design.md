# SKILL.md Orchestration Cache Integration Design

**Date:** 2026-03-30
**Status:** Approved (2 architecture review rounds + 2 spec review rounds, converged)
**Repo:** `ugiordan/adversarial-review`
**Depends on:** `docs/specs/2026-03-27-local-context-cache-design.md` (foundation spec)

## Summary

Integrate the local context cache (`manage-cache.sh`) into the adversarial-review orchestration pipeline. Cache becomes the default and only dispatch mode — eager loading (injecting all context into agent prompts) is removed. Agents read code, templates, references, and findings from a temporary disk cache via the Read tool.

This design covers SKILL.md, `phases/self-refinement.md`, and `phases/challenge-round.md` changes, plus flag handling for `--keep-cache`, `--reuse-cache`, and `--delta`.

## Approach

**Cache as default, eager loading removed.** The cache foundation is tested (230 tests, ShellCheck clean). Maintaining two parallel dispatch paths in a rigid skill creates confusion. If shell execution is unavailable, the existing SKILL.md fallback ("fall back to LLM-based validation with disclaimer") already covers degraded operation.

## Section 1: Phase 0 — Cache Initialization

A new step inserted into SKILL.md between scope confirmation (current Step 2) and Phase 1 dispatch (current Step 3). All subsequent steps renumber (+1).

### New Step 3: Initialize Cache

After scope is confirmed and before any agents spawn:

1. **Generate session hex:** `openssl rand -hex 16` — identifies this cache session. Independent from delimiter generation (different value, different purpose).
2. **Initialize cache:** `manage-cache.sh init <session_hex>` — creates cache directory, lock file, manifest. Capture `CACHE_DIR` from JSON stdout.
3. **Generate delimiter hex:** `scripts/generate-delimiters.sh` — produces session-wide `REVIEW_TARGET` delimiter hex for code wrapping. Collision-checked against all scope files. This is a separate 128-bit hex from the session hex.
4. **Populate code:** `manage-cache.sh populate-code <scope_file> <delimiter_hex>` — copies scope files with delimiter wrapping + anti-instruction text.
5. **Populate templates:** `manage-cache.sh populate-templates` — copies finding + challenge response templates.
6. **Populate references:** `manage-cache.sh populate-references` — copies enabled reference modules.
7. **Generate navigation:** `manage-cache.sh generate-navigation 1 1` — creates `navigation.md` for iteration 1, phase 1.
8. **Set cleanup trap:** The orchestrator runs via Bash: `trap "CACHE_DIR='$CACHE_DIR' '$SCRIPT_DIR/manage-cache.sh' cleanup" EXIT HUP INT TERM` (using absolute path to `manage-cache.sh`). This ensures cache cleanup on any exit. **Skipped if `--keep-cache` is specified.**
9. **Export `CACHE_DIR`** for all subsequent steps.

**Failure behavior:** If any step 2-6 fails, abort the review with error. No fallback to eager loading. See Section 5 for full error table.

### Session Hex vs. Delimiter Hex

| Value | Purpose | Generated by | Used in |
|-------|---------|-------------|---------|
| Session hex | Cache directory naming, `--reuse-cache` resolution | `openssl rand -hex 16` | `manage-cache.sh init`, `last-cache.json` |
| Delimiter hex | Code file wrapping (`REVIEW_TARGET` markers) | `scripts/generate-delimiters.sh` | `manage-cache.sh populate-code`, agent prompts |

Both are 128-bit CSPRNG random hex values. They are generated independently and serve different purposes.

### Session-Wide Delimiter Relaxation

In non-cache mode, `protocols/input-isolation.md` specifies per-agent delimiter pairs. In cache mode, a single session-wide delimiter hex is used for `REVIEW_TARGET` code wrapping to avoid duplicating every code file per agent. This is an accepted trade-off:
- The hex is 128-bit CSPRNG random
- Collision-checked against all scope files
- Agents don't communicate directly
- `FIELD_DATA` markers in sanitized findings retain per-field unique hex values (unchanged)

`protocols/input-isolation.md` must be updated to document this behavioral difference.

## Section 2: Phase 1 — Minimal Prompt Dispatch

Replaces the eager loading dispatch in `phases/self-refinement.md` Steps 1-2.

### Agent Prompt Composition

Each agent receives a minimal prompt containing:

| Content | Source | ~Tokens |
|---------|--------|---------|
| Role definition | `agents/<specialist>.md` (unchanged) | ~1,500 |
| Inoculation paragraphs (3) | Already in role files (unchanged) | ~500 |
| Delimiter values | Session-wide hex | ~125 |
| Finding template | `templates/finding-template.md` inline | ~500 |
| Cache navigation block | Inline text (see below) | ~200 |

**Total: ~2,825 tokens per dispatch** (down from 15K-50K with eager loading).

### Cache Navigation Block (In Prompt)

```markdown
## Cache Access

Your review materials are at: {CACHE_DIR}

Read `{CACHE_DIR}/navigation.md` FIRST — it tells you what's available and what to read.

## Mandatory Reads
Read these files before producing findings:
- {CACHE_DIR}/code/{file1}
- {CACHE_DIR}/code/{file2}
- ...

Rules:
- Read code files from `code/` before making claims about them
- In Phase 2: read `findings/cross-agent-summary.md` first, then full finding files only for findings you challenge or that fall in your domain
- You MUST Read the full finding before issuing a Challenge
- Use repo-relative paths in findings (e.g., `src/auth/handler.go`), not cache paths
- Read references on iteration 2+
```

The mandatory reads list includes all scope files, ensuring agents read code even if they skip `navigation.md`.

### Between-Iteration Flow

Replaces `self-refinement.md` Steps 3-7:

1. **Collect agent output** — gather raw output from each agent.
2. **Validate + sanitize + split:** `manage-cache.sh populate-findings <agent> <role_prefix> <output_file> --scope <scope_file>` — this single call replaces the separate `validate-output.sh` invocation. It validates (with scope check), applies sanitized document template, splits into individual finding files, and generates the summary table. **Note:** The `--scope` parameter does not exist in the current implementation — see Section 7 prerequisite #1 for the required change.
3. **Detect convergence:** `scripts/detect-convergence.sh` (unchanged).
4. **Update navigation:** `manage-cache.sh generate-navigation <iteration+1> 1` — regenerates `navigation.md` with updated iteration number.
5. **Re-prompt agent** with self-refinement instructions + verification gate (iteration 2+) + reference cross-check (iteration 2+). The agent reads its own prior findings from `{CACHE_DIR}/findings/<agent>/sanitized.md` in cache. The orchestrator no longer feeds prior output back into the prompt.

### What Stays In Self-Refinement Re-Prompt

The re-prompt contains only:
- Self-refinement instruction: "Review your prior findings at `{CACHE_DIR}/findings/<agent>/sanitized.md`. What did you miss? What's a false positive? Refine."
- Verification gate text (iteration 2+, unchanged from current)
- Reference cross-check text (iteration 2+, unchanged from current)
- Note: "Read reference modules from `{CACHE_DIR}/references/` for cross-checking."

## Section 3: Phase 2 — Challenge Round with Selective Reading

Replaces the full sanitized document injection in `phases/challenge-round.md` Steps 1-4.

### Before Dispatching Challengers

1. **Pre-debate deduplication:** `scripts/deduplicate.sh` (unchanged).
2. **Build cross-agent summary:** `manage-cache.sh build-summary` — merges all agent `summary.md` files into `findings/cross-agent-summary.md`.
3. **Generate Phase 2 navigation:** `manage-cache.sh generate-navigation <iteration> 2 --resolved-ids <resolved_file>` — updates `navigation.md` for Phase 2. Resolved findings (from prior challenge iterations) are omitted from the listing. **Note:** The `--resolved-ids` parameter does not exist in the current implementation — see Section 7 prerequisite #2 for the required change and expected behavior.
4. **Generate finding IDs file:** Extract the ID column from `cross-agent-summary.md` into a file (one ID per line) for `validate-output.sh --mode challenge --finding-ids`.

### Challenge Prompt Composition

Same minimal prompt structure as Phase 1 (~2,825 tokens) with Phase 2 cache navigation:

```markdown
## Cache Access — Phase 2

Your review materials are at: {CACHE_DIR}

Read `{CACHE_DIR}/navigation.md` FIRST.

1. Read `findings/cross-agent-summary.md` — overview of all findings
2. Read full finding files ONLY for findings you intend to challenge or that fall in your domain
3. You MUST Read the full finding before issuing a Challenge — you cannot challenge based on the summary alone
4. Use `templates/challenge-response-template.md` for your response format
```

### Two-Tier Finding Access

**Tier 1 — Summary table:** Agent reads `findings/cross-agent-summary.md` (~200 tokens). Contains: ID, Severity, Category, File:Line, One-liner for every finding.

**Tier 2 — Full details on demand:** Agent reads individual finding files (`findings/<agent>/<ID>.md`) only for findings they intend to challenge or that fall in their domain.

**Token impact:** ~20K per agent (eager loading) → ~200 tokens (summary) + ~2,500-5,000 tokens (selective reads). Savings depend on challenge rate.

### Validation Pipeline (Phase 2)

Two distinct validation paths:

1. **Challenge responses:** Orchestrator calls `validate-output.sh --mode challenge --finding-ids <ids_file>` directly (not through `populate-findings`).
2. **New findings raised during challenge:** Orchestrator calls `manage-cache.sh populate-findings <agent> <role_prefix> <output_file> --scope <scope_file>` (same as Phase 1 between-iteration flow).

### Between Challenge Iterations

1. Collect output.
2. Validate challenge responses (path 1 above).
3. Process new findings if any (path 2 above).
4. Identify resolved findings (all-Agree) — add their IDs to the resolved file.
5. `manage-cache.sh generate-navigation <iteration+1> 2 --resolved-ids <resolved_file>` — regenerates navigation with resolved findings omitted.
6. Re-prompt with updated challenge context.

### Context Cap Enforcement

`navigation.md` includes token estimates per file (char/4 heuristic). If total cache content exceeds the 50K per-iteration context cap:
- Navigation lists files in severity order (Critical findings first)
- Notes: "N findings omitted due to context limits"
- Omitted findings carried forward to resolution with Phase 1 status
- Agents naturally read highest-priority items first

## Section 4: Flag Handling — `--keep-cache`, `--reuse-cache`, `--delta`

### `--keep-cache`

**Phase 0 change:** Skip the cleanup trap (step 8 in Section 1).

**After Phase 4 (report):** Write `.adversarial-review/last-cache.json` to repo root:
```json
{"session_hex": "<hex>", "commit_sha": "<HEAD>"}
```

**User output:** Print session hex: "Cache preserved. Reuse with `--reuse-cache <hex>`"

`.adversarial-review/` is already in `.gitignore`.

### `--reuse-cache <hex>`

**Phase 0 replacement:** Steps 2-7 in Section 1 are replaced with:

1. Validate hex argument: must match `^[a-f0-9]{32}$` (same validation as `manage-cache.sh init`). Then scan `$TMPDIR` for directories matching `adversarial-review-cache-<hex>-*`.
2. `manage-cache.sh validate-cache <path>` — SHA-256 per-file + commit SHA against current HEAD.
3. If valid: set `CACHE_DIR`, skip all populate steps.
4. If invalid: abort with error listing mismatches. Do NOT silently create a new cache.
5. Clear findings directory — `rm -rf "$CACHE_DIR/findings/"*` then `mkdir -p "$CACHE_DIR/findings"`. Findings are regenerated from scratch (new review, same code).
6. Regenerate navigation: `manage-cache.sh generate-navigation 1 1`.

### `--delta`

**Before Phase 0:** Check for `.adversarial-review/last-cache.json` in repo root.

- **If found:** Display session hex + commit SHA to user. Ask for explicit confirmation to reuse.
- **If confirmed:** Behave as `--reuse-cache <hex>` (same flow above).
- **If declined or not found:** Create new cache normally (full Phase 0).

### Flag Interaction Matrix

| Combination | Behavior |
|------------|----------|
| `--delta` alone | Auto-discover previous cache via `last-cache.json`, prompt user, new cache if declined/not found |
| `--reuse-cache <hex>` alone | Resolve specific cache, validate, reuse code/templates/refs, regenerate findings |
| `--delta` + `--reuse-cache` | **Mutually exclusive.** Error: "Use --delta for auto-discovery or --reuse-cache for explicit reuse, not both." |
| `--delta` + `--keep-cache` | Composable. Reuses previous cache if confirmed, keeps the new/reused cache after completion. |
| `--reuse-cache` + `--keep-cache` | Composable. Reuses specified cache and preserves it after completion. |
| `--diff` + `--reuse-cache` | **Mutually exclusive.** Error: "--diff creates a minimal cache from changed files; --reuse-cache expects a complete cache." |
| `--diff` + `--delta` | Composable. Delta discovers previous cache for comparison; diff limits scope to changed files. |

### `last-cache.json` Implementation Note

Both the write (after `--keep-cache`) and read (during `--delta`) of `last-cache.json` are orchestrator-level behaviors implemented in SKILL.md markdown instructions. No backing script is needed — the orchestrator writes/reads the JSON file directly.

## Section 5: Error Handling Changes

New cache-related failure modes added to SKILL.md's error handling table.

### Cache Errors

| Scenario | Response |
|----------|----------|
| `manage-cache.sh init` fails | Abort review with error. No fallback to eager loading. |
| `populate-code` fails (collision, missing file, path traversal) | Abort review — cache integrity cannot be guaranteed. |
| `populate-templates` fails | Abort review — agents need templates for format compliance. |
| `populate-references` fails | Abort review — reference modules required for cross-check. |
| `populate-findings` fails (validation or injection) | Same as current: spawn fresh agent with error message, max 2 retries per agent per iteration. If retries exhausted, exclude agent's findings for this iteration and log failure. |
| `validate-cache` fails on `--reuse-cache` | Abort with mismatch details. User must create new cache or address the issue. |
| `generate-navigation` fails | Non-fatal warning. Agents can still Read code files directly from `code/` using the mandatory reads list in their prompt. Log warning, continue. |
| Cache directory disappears mid-review | Abort review — unrecoverable. Likely external cleanup or disk issue. |
| `last-cache.json` missing/corrupt on `--delta` | Treat as "not found" — create new cache normally, inform user. |

### Design Principle

Cache initialization and population failures are **fatal** (abort review). Between-iteration failures (`populate-findings`, `generate-navigation`) use **existing retry/exclusion logic** — the review can continue with degraded data.

## Section 6: Phase 3/4/5 Cache Interaction

### Phase 3 (Resolution)

No changes. The orchestrator collects finding data during Phases 1-2 as it processes agent outputs: each call to `populate-findings` returns status, and the orchestrator tracks finding IDs, severities, and challenge positions through its own state (the LLM orchestrator maintains this across turns in its conversation context). `deduplicate.sh` operates on files passed as arguments (the orchestrator can read from `findings/*/sanitized.md` if needed). The cache serves as persistent storage; the orchestrator's conversation context serves as working memory for resolution logic.

### Phase 4 (Report)

No changes. The report is generated by the orchestrator from the validated finding set produced by Phase 3. Cache data is not read during report generation.

### Phase 5 (Remediation)

No changes. Remediation (`--fix`) works from the final finding set. Worktrees and fixes operate on the actual repo, not the cache.

### Cleanup Timing

The cleanup trap fires on EXIT after all phases complete. If `--keep-cache` is specified, the trap is not set and the cache persists for future `--reuse-cache` or `--delta` use.

## Section 7: Implementation Prerequisites

Changes needed in files other than SKILL.md and phase files:

### Script Changes

1. **`manage-cache.sh` populate-findings:** Add `--scope <file>` optional parameter. When provided, pass through to `validate-output.sh` as `--scope <scope_file>`. Current implementation (line 241) calls `validate-output.sh` without scope — this must be extended.

2. **`manage-cache.sh` generate-navigation:** Add `--resolved-ids <file>` optional parameter. When provided, read finding IDs from the file (one per line). During navigation generation, omit any finding whose ID appears in this list from the findings summary section of `navigation.md`. The `generate-navigation` Python code should load the resolved set, then filter summary rows before writing. Resolved findings remain in the cache (for audit) but are not surfaced to agents.

3. **`manage-cache.sh` usage header (lines 4-11):** Add `generate-navigation <iteration> <phase>` to the usage comment block. Currently missing from documentation.

4. **`manage-cache.sh` generate-navigation:** Add context cap enforcement. When total token estimate across all listed files exceeds 50K, list files in severity order (Critical first) and append a note: "N findings omitted due to context limits." This logic does not exist in the current implementation.

5. **`validate-output.sh`:** Add cache-path prefix stripping as fallback. If a finding references a file path starting with a known cache prefix pattern (`*/adversarial-review-cache-*/code/`), strip the prefix to recover the repo-relative path. Emit a warning when stripping occurs.

### Documentation Changes

6. **`protocols/input-isolation.md`:** Add note documenting session-wide delimiter relaxation: "When cache mode is active, a single session-wide `REVIEW_TARGET` delimiter hex is used across all agents. This avoids duplicating every code file per agent in the cache. `FIELD_DATA` markers in sanitized findings retain per-field unique hex values. See the local context cache spec (Section 3) for rationale."

7. **Naming standardization:** The foundation spec (`2026-03-27-local-context-cache-design.md`) uses `CACHE_PATH` in some places (Sections 2, 10). Normalize to `CACHE_DIR` to match the implementation. The spec's Section 9 subcommand table also documents `populate-findings` with a 2-argument signature — the actual implementation takes 3 arguments (`<agent> <role_prefix> <findings_file>`). These are spec errata to correct during implementation.

## Known Limitations

1. **Budget tracking undercount:** Tokens consumed by agent Read calls (reading from cache) are not tracked by `track-budget.sh add`. The pre-flight estimate accounts for this via the `prompt_overhead` term, but per-iteration actual tracking only counts agent output characters. The spec acknowledges this limitation.

2. **Triage mode not addressed:** `--triage` has its own validation script and convergence criteria. Cache integration for triage mode is deferred to a follow-up design.

3. **Single-specialist devil's advocate:** The devil's advocate agent in Phase 2 single-specialist mode uses the same cache navigation pattern (reads `cross-agent-summary.md` then individual findings). No special handling needed — the cache structure is agent-agnostic.

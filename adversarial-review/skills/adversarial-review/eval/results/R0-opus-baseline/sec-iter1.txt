File: internal/controller/services/auth/auth_controller_actions.go
Lines: 63-65
Title: Operator precedence bug in system:authenticated guard allows bypassing admin group protection
Evidence: In the bindRole function, the condition to skip system:authenticated from admin roles is: if roleName == "admingroup-role" && e == "system:authenticated" || e == "". Due to Go operator precedence, && binds tighter than ||, so this evaluates as: (roleName == "admingroup-role" && e == "system:authenticated") || e == "". This means system:authenticated is ONLY blocked from the "admingroup-role" role. The same bug exists in bindClusterRole. The AllowedGroups path uses role name "allowedgroup-role", so system:authenticated passes through without filtering. The operator precedence is technically a bug that would matter if the code were refactored.
Recommended fix: Add parentheses for clarity and correctness. Better yet, refactor the "skip system:authenticated for admin roles" logic into a dedicated function.

Finding ID: SEC-002
Specialist: Security Auditor
Severity: Important
Confidence: High
Source Trust: Internal
File: pkg/cluster/cert.go
Lines: 106-118
Title: Self-signed certificate generated as CA with overly permissive key usage
Evidence: In generateCertificate(), the self-signed TLS certificate template includes KeyUsage: x509.KeyUsageCertSign and IsCA: true. This means the generated certificate can be used to sign other certificates, which is unnecessary for a server TLS certificate. If the private key is compromised, an attacker could use it to mint arbitrary certificates. The certificate is stored in a Kubernetes Secret and used for TLS on the gateway.
Recommended fix: Remove x509.KeyUsageCertSign from KeyUsage and set IsCA: false for server certificates. Generate a separate CA certificate if needed.

Title: Fallback container image uses :latest tag from personal repository
Evidence: In getKubeAuthProxyImage(), the function returns "quay.io/jtanner/kube-auth-proxy:latest" as a fallback when the RELATED_IMAGE_ODH_KUBE_AUTH_PROXY_IMAGE environment variable is not set. Finding ID: SEC-004
Specialist: Security Auditor
Severity: Important
Confidence: Medium
Source Trust: Privileged
File: internal/controller/services/gateway/gateway_auth_actions.go
Lines: 55-88
Title: OAuth2 proxy configured with --email-domain=* allowing any email domain
Evidence: In buildBaseOAuth2ProxyArgs(), the OAuth2 proxy is configured with --email-domain=*, which disables email-domain-based access restriction. Combined with --skip-jwt-bearer-tokens=true, bearer tokens can bypass the OAuth login flow. For a data science platform, any authenticated cluster user could access the gateway.
Recommended fix: Consider restricting --email-domain to specific organizational domains. Make the email domain configurable via the GatewayConfig CRD.

Finding ID: SEC-005
Specialist: Security Auditor
Severity: Important
Confidence: Medium
Source Trust: Privileged
File: internal/controller/services/gateway/gateway_support.go
Lines: 30-32
Title: OAuth scope user:full grants excessive permissions to the gateway OAuth client
Evidence: OpenShiftOAuthScope is set to "user:full", which grants the OAuth client full access to all APIs that the user has access to, including impersonation. A more restrictive scope like user:info or user:check-access would be sufficient if the gateway only needs to verify identity and group membership.
Recommended fix: Reduce the OAuth scope to user:info or user:info user:check-access.

Title: Gateway listener allows routes from all namespaces with no restrictions
Evidence: In CreateListeners(), the Gateway is configured with from := gwapiv1.NamespacesFromAll, meaning HTTPRoutes from ANY namespace can attach to this gateway. This could allow a user in a non-privileged namespace to expose their service through the data science gateway.
